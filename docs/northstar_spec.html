0. Project context

Project North Star is an AI-powered wellness platform for teenagers (ages 13–18). The product is being built by Boost AI for a client (Offy, the funder) whose son (Lev, age 16) is the product visionary. The platform is web-first, architected for an iOS port later.

The audience is teenagers under significant pressure from social media, academic life, social dynamics, and identity formation. The product is not therapy. It is a structured coaching environment that helps teens take action on their wellness, discover their strengths, and feel less alone. The voice of the product is warm, real, slightly irreverent, never preachy, and never makes a teenager feel small.

Core product loop

  1. A teenager signs up. With parental consent if under 18.
  2. They meet Kai, the AI mentor. They can rename Kai (Coach, Buddy, Pal, anything) and lightly customize tone.
  3. Kai asks a short series of questions to understand what the teen is dealing with right now.
  4. Based on the answers, Kai routes the teen into one of three engines: Physical Wellness, Potential & Goals, or Mental Wellness.
  5. The teen engages with that engine — content, exercises, reflections, tracking — and Kai checks in regularly.
  6. A cross-cutting progress tracker visualizes growth: personal charts, an evolving character avatar, and opt-in comparison with friends.
  7. Over time, the teen can engage all three engines and use Kai as their general-purpose mentor.

1. Editorial guardrails (read first)

These rules govern every line of content, every Kai response, every UI string. Apply them ruthlessly.

Voice

  • Warm, real, slightly irreverent. Sound like a cool older sibling, not a guidance counselor.
  • No preaching. Never tell a teen they "should" feel a certain way.
  • No corporate language. No "leverage," "synergy," "transform your life."
  • Short sentences. Active voice. Concrete nouns.
  • Plain language. Grade 8 for content, grade 6 for UI.
  • No emoji by default. Teens read excessive emoji as cringe.

Things Kai never does

  • Never diagnoses anything. Not depression, not anxiety, not anything.
  • Never prescribes medication, supplements, dosages, or specific drugs.
  • Never tells a teen what's wrong with them. Asks questions, reflects what it hears, offers options.
  • Never claims to be human.
  • Never agrees with self-harm, suicide, eating-disorder behavior, substance abuse, or violence — even softly, even rhetorically.
  • Never shares any user's data with another user.
Crisis content — hard rules

If a user expresses suicidal ideation, self-harm, active eating disorder behaviors, disclosure of abuse, acute substance abuse, or threats to harm others — the safety layer takes over immediately. Kai does not coach. Kai hands off. See Section 7 for the full safety architecture.

The safety classifier (runs on every message)

Every inbound user message runs through a safety classifier before being passed to the main model. The classifier returns:

{
  "categories": ["suicide_ideation" | "self_harm" | ... | "none"],
  "severity": "low" | "medium" | "high" | "critical",
  "explanation": "<short reason>"
}

If any category is non-none at medium severity or higher, the main model is bypassed. The safety layer takes over.

The three engines

Physical Wellness

Domain: Nutrition, exercise, sleep, breathing, yoga, stretching. The body as the foundation of how a teenager feels day to day. Build habits, pursue physical goals without falling into diet culture or appearance obsession.

Grounded in: Whole-food-first nutrition (no calorie obsession), movement they enjoy (not punishment), sleep as infrastructure, breathwork as daily practice, yoga as nervous-system regulation.

Never does: Calorie obsession, supplement recommendations, body comparison, "no pain no gain" framing, food moralizing, eating disorder diagnosis (safety layer takes over).

Potential & Goals

Domain: Discovering hidden strengths and pursuing real goals. School, instruments, sports, business, charity, creative work. Help teens notice what they're naturally good at, set goals that matter to them, and stay with it through the hard middle.

Grounded in: Strengths-based discovery, goals specific enough to act on and modest enough to start, process over outcome, self-determination over external validation, failure as data.

Never does: Tell teens what they "should" do, compare to peers/siblings, encourage parent-pleasing goals, treat failure as failure, push entrepreneurship as inherently superior.

Mental Wellness

Domain: Self-esteem, identity, emotion regulation, nervous-system literacy, specific pressures of social media. Help teens name feelings, understand body and mind responses, build practices. Not a therapist. Says so clearly.

Grounded in: Source materials to be selected with Offy and Lev. Nervous-system literacy, identity formation, anti-comparison framing, breath and body as primary regulation tools.

Never does: Diagnose, tell feelings are wrong, push positive thinking on pain, replace therapy. Safety layer priority: HIGH.

7. The safety layer

The safety layer is not a feature. It is infrastructure. It exists at four levels.

Level 1: Inbound message classifier

Every message a user sends, in any engine, runs through a fast classifier before the main model sees it.

Level 2: The safety response

When the safety layer takes over, the user sees:

  1. A warm acknowledgment generated by claude-opus-4-7, streamed in immediately
  2. A clear "this is bigger than I can help with directly" statement
  3. Crisis resources rendered in a card: 988, Crisis Text Line, Trevor Project, NEDA helpline
  4. A parent-notification offer if user is under 18 and parent email is on file
  5. A "stay with me" prompt — the conversation does not end

Level 3: The crisis page

/crisis is a static page accessible from every footer, always reachable, no login required. Loads in under 1 second. Works offline (PWA cache).

Level 4: Operations review

Every safety event is logged to safety_events. An ops dashboard lets a designated operator review all events daily. For critical severity, an immediate email goes to the operations email address — user ID only, no raw user text.

Parental notification — when and how

Always offer first. System asks "is it OK if I let your parent know you reached out?" before notifying. Always notify for critical severity in suicide/self-harm/abuse categories, regardless of user consent, if parent contact is on file. This is named explicitly in Terms of Service so it's not a surprise.

Notification content: a calm email saying their child reached out about a serious topic, that crisis resources were provided, and that they should reach out to their teen. No raw chat content. No specifics.

8. Onboarding flow

The onboarding flow is the most important sequence in the entire product. It is where trust is built or lost. Every step has been designed deliberately. Build it exactly as specified.

Step 0: Landing page (pre-signup)

A single page at /. Hero: "An AI mentor for the stuff teen life actually throws at you." Three-card explanation of Kai, the engines, and the safety promise. A "For parents" button. Single CTA: "Get started." No marketing fluff. No fake testimonials.

Step 1: Account creation

Via Clerk. Email + password, or Google/Apple OAuth. Phone optional. Capture age immediately after.

Step 2: Parental consent (if under 18)

  • Ask for parent's email
  • Send parent a one-click consent email via Resend
  • Teen sees "while you wait" screen with crisis resources prominently displayed
  • Teen cannot proceed until parent has clicked the consent link
  • Parent's consent action is logged in users.parent_consent_at

Step 3: Basics

Three short questions:

  1. What's your first name? (or nickname)
  2. How old are you? (confirm)
  3. Anything you want me to know up front? (optional, freeform)

Step 4: Meet Kai (customization)

  1. What do you want to call your mentor? Default: "Kai". Free text, up to 20 chars.
  2. How direct do you want them to be? Warm / Balanced / Direct.
  3. Voice preview: Canned sample messages in each tone, swipeable.

Step 5: Intake questions

Six questions, asked one at a time:

  1. Walk me into a normal day for you. What does it look like from when you wake up to when you go to bed?
  2. What's one thing you wish was different about your life right now?
  3. What's one thing you actually like about your life right now?
  4. Where do you feel pressure these days — and where's it coming from?
  5. If you could spend an extra hour every day on anything, no judgment, what would you do with it?
  6. On a scale of 1 to 10, how are you actually doing this week?

After Q6, Kai writes a 3-sentence summary (using claude-haiku-4-5) that becomes part of every future context.

Step 6: Engine routing

Kai routes the teen to a primary engine using a structured-output call. The teen sees: "Based on what you shared, let's start with [Engine]. Here's why: [Kai's one-sentence reasoning]. You can switch any time."

Two buttons: "Sounds good. Let's start." or "Let me pick something else." (shows all three).

Step 7: First real conversation

Teen lands in their chosen engine. Kai opens with engine-specific opening style. Intake summary is in Kai's context.

Step 8: The character is born

After the first conversation (5+ messages or 5+ minutes), the evolving character avatar is generated at Level 1. Teen sees brief animation: "This is you. As you use this, you'll see them grow."

9. The cross-cutting progress tracker

Three views on the /progress page:

1. Personal chart

Line/area chart showing wellness contribution over last 30 days. Smoothed daily rollup. Color-coded by engine.

3. Friend compare

Opt-in. Requires mutual accept. Shows streak, level, weekly score only. Never shows conversation content, goal content, reflection content, or meal photos. Aggregate stats only.

Karate-belt achievement system

Lev's idea. Belts for engine-specific milestones (white → yellow → green → blue → black). Visible on user profile and to friends.

10. The food-photo nutrition flow

The most engineered feature in v1.

  1. User taps "Log a meal" anywhere in the physical engine
  2. Takes photo or uploads one
  3. Photo uploaded to R2
  4. Frontend calls POST /api/food-photo
  5. Worker calls Cloudflare Workers AI vision with structured prompt → returns JSON of food items with estimated grams
  6. For each item, Worker calls USDA FoodData Central API for nutrition data
  7. Sums totals, saves to meals table
  8. Returns breakdown to frontend with edit option
Critical: framing the response

Never show the meal as "good" or "bad." Never compare to a daily target unless the user has explicitly asked to track macros. Default presentation is descriptive, not evaluative.

12. The five phases

Each phase is a payment gate. Each phase ends with a working demo. Phases run sequentially.

PHASE 1

Foundations & Kai

  1. Initialize repo, configure Cloudflare (Pages, Workers, D1, KV, R2)
  2. Set up Clerk with age-aware sign-up
  3. Build design system per Section 4
  4. Build app shell, nav, footer (footer MUST include crisis link)
  5. Build always-on /crisis page
  6. Build landing page
  7. Build onboarding flow per Section 8
  8. Build safety classifier (Section 7 Level 1+2)
  9. Build Kai chat surface with Section 6 system prompt
  10. Build progress tracker scaffold, character Level 1 only
  11. Write Terms of Service and Privacy Policy
Gate G2: Teen can sign up, complete consent, meet Kai, customize, complete intake, be routed. Safety is live.
PHASE 2

Physical Wellness Engine

  1. Pick source materials with Offy and Lev
  2. Build engine page with Kai as host
  3. Build food-photo flow end-to-end (Section 10)
  4. Build workout, sleep, breathing, yoga loggers
  5. Wire events to progress_events
  6. Connect to character (now through Level 3 minimum)
  7. Friend-test with 5+ teens
Gate G3: All physical features work end-to-end. Food-photo is accurate enough to be useful. Character visibly evolves.
PHASE 3

Potential & Goals Engine

  1. Pick source materials
  2. Build engine page
  3. Build strengths-discovery flow (15-min guided sequence)
  4. Build goal CRUD across all categories
  5. Build celebration animations
  6. Build goal-reframe conversation
  7. Wire events, connect to character, add belts
  8. Friend-test with 5+ teens
Gate G4: Strengths discovery works. Goals can be set, tracked, completed. Belts award correctly.
PHASE 4

Mental Wellness Engine + Safety Hardening

Safety first, engine second.

  1. Harden safety classifier based on Phases 1–3 learnings
  2. Build parent-notification flow (Section 7 Level 4)
  3. Build ops review dashboard
  4. Get third-party clinical review (provided by Offy) BEFORE engine launch
  5. Build mental engine page
  6. Build feelings check-in, contextual breathing, meditation, social-media reset, thought reframe, letter writing
  7. Wire events, connect to character, add belts
  8. Friend-test ALL safety-flagged for review
Gate G5: Mental engine live, clinically reviewed, integrated with hardened safety layer. No unhandled safety incidents.
PHASE 5

Integration, Testing & Launch Prep

  1. Full cross-engine integration testing
  2. Performance testing: 100+ concurrent users, <2s loads
  3. Accessibility audit: WCAG AA
  4. Real-user testing with 15–20 teens, sustained usage
  5. Critical bug triage and fix
  6. Build analytics dashboard
  7. Write user-facing help docs
  8. Write internal handoff docs
  9. Soft launch plan locked in
  10. Final production deploy on chosen custom domain
Gate G6: Platform is live, tested, accessible, ready for real users. Documentation handed off.

13. Agent execution notes

Order of operations

  1. Read this entire document before writing any code. Re-read sections 1, 7, 8.
  2. Initialize the repo structure (Section 3) first.
  3. Build the safety classifier (Section 7 Level 1) before any conversational surface. Non-negotiable.
  4. Build the onboarding flow (Section 8) before any engine.
  5. Build engines in Section 12 order, not in parallel.

Conventions

  • Always-on safety: Every chat code path calls the safety classifier before the main model. No exceptions.
  • All prompts in workers/src/lib/prompts/. Never inline system prompts.
  • All AI calls go through workers/src/lib/claude.ts. Never call Anthropic SDK from routes.
  • Errors logged with user ID + request ID. No raw user text in logs.
  • Never commit .env.

What NOT to do

  • Don't introduce new dependencies without asking.
  • Don't build engines in parallel.
  • Don't skip the safety classifier "for testing."
  • Don't add features not in this spec.
  • Don't introduce React Native components in v1.
